# Prompts para mockups realistas (Gemini / Nano Banan)

Objetivo: generar imágenes que parezcan **vistas previas reales** de lo que el cliente tendrá, con **dispositivos realistas** (notebook y celular que se vean de verdad) y una **UI neutra** para poder **superponer logo y colores** del cliente sin que se note el pegote.

---

## Reglas que deben cumplir TODOS los prompts

1. **Dispositivo realista**
   - **Escritorio:** una **notebook/laptop real** en la escena (marco, teclado, bisagra), vista en 3/4 o frontal, con la pantalla mostrando la interfaz. No solo una pantalla flotante.
   - **Móvil:** un **smartphone real** (marco, bordes, notch o isla si aplica), en vertical, opcionalmente sostenido por una mano. La pantalla debe mostrar la UI.
2. **UI preparada para logo y color**
   - **Barra superior** o header en un solo color **gris muy claro o blanco** (o un color sólido suave), **sin logo ni texto de marca**, para que luego se superponga el logo del cliente.
   - **Un solo color de acento** en toda la interfaz (botones, enlaces, iconos activos): por ejemplo **#888** o **rojo muy suave**. Ese color se podrá reemplazar por el color primario del cliente.
   - Resto en grises neutros (fondos #f5f5f5, #fff, texto #333).
3. **Elementos específicos cuando aplique**
   - Si es web: incluir **botón flotante de WhatsApp** (círculo verde con ícono de WhatsApp) en una esquina, para que el cliente lo vea.
   - Si es catálogo: **grid de productos** con imagen, nombre, precio y botón.
   - Si es sistema con roles: en la UI se puede sugerir **“Admin”**, **“Técnico”**, **“Sucursal”** en menú o filtros.
4. **Sin marcas ni textos reales**
   - Textos de relleno genéricos (“Nombre del negocio”, “Menú”, “Producto 1”) o Lorem. Nada de logos reconocibles ni nombres de empresas reales.

## 1. WEB – Tipos: landing, multi‑página, catálogo

### 1.1 Web tipo landing (una página) – escritorio en notebook real

> Fotografía o render **realista** de una **notebook/laptop** (marco, teclado, pantalla) mostrando una **página web tipo landing de un negocio**: barra superior **en un solo color gris claro o blanco, sin logo**, menú con 3–4 ítems (Inicio, Servicios, Contacto), hero con título grande y un botón principal (ej. “Contactar”), debajo 3 tarjetas de servicios con ícono y texto corto. En una esquina inferior derecha de la **pantalla dentro del notebook**, un **botón flotante redondo verde de WhatsApp**. Colores de la web: fondos blancos/gris muy claro, textos grises oscuros, botones y el ícono de WhatsApp en un **único color de acento** (verde WhatsApp o un rojo/azul suave) para poder recolorear después. Estilo moderno y limpio. La notebook debe verse como un equipo real, no una caja plana.

### 1.2 Web tipo landing – móvil en celular real

> Mockup **realista** de un **smartphone** (marco negro o gris, bordes, posible notch) en **vertical**, mostrando la **versión móvil de una landing**: barra superior con espacio vacío para logo y menú hamburguesa, hero con título y botón, bloque de servicios en tarjetas apiladas. En la **esquina inferior derecha de la pantalla**, **botón flotante verde de WhatsApp**. Paleta neutra (blanco, gris claro, un solo color de acento para botones). El teléfono debe parecer un dispositivo real, opcionalmente sostenido por una mano.

### 1.3 Web multi‑página (varias secciones) – escritorio en notebook

> **Notebook realista** con pantalla que muestra un **sitio web de varias secciones**: header con barra **gris clara sin logo**, menú (Inicio, Nosotros, Servicios, Galería, Contacto), contenido central con título de sección (ej. “Nuestros servicios”) y grid de 4–6 tarjetas con ícono y texto. **Botón flotante de WhatsApp** en esquina inferior derecha de la pantalla. Un solo color de acento en botones y enlaces. Fondos y textos neutros. El laptop debe verse real (marco, teclado).

### 1.4 Web con catálogo online – escritorio en notebook

> **Laptop realista** con pantalla mostrando una **página de catálogo de productos**: barra superior neutra sin logo, menú con “Inicio”, “Catálogo”, “Contacto”, zona de **grid de productos** (imagen placeholder, nombre, precio, botón “Ver” o “Agregar”). Filtros laterales o arriba (categoría, orden). **Botón flotante de WhatsApp** visible en una esquina. Un único color de acento para botones. Todo lo demás en grises y blanco para poder superponer logo y color de marca.

### 1.5 Web con catálogo – móvil en celular real

> **Smartphone real** en vertical mostrando **catálogo en móvil**: header con espacio para logo, barra de búsqueda, listado de productos en tarjetas (foto, nombre, precio). **Botón flotante de WhatsApp** en esquina. Paleta neutra con un color de acento. Dispositivo realista.

### 1.6 Sección interna (ej. página de servicio o contacto) – notebook

> **Notebook realista** con pantalla de **página interna** de un sitio: misma barra superior neutra, menú, contenido central con título (ej. “Contacto”), formulario o bloque de texto y **botón flotante de WhatsApp**. Un color de acento, resto neutro.

## 6. Cómo usar estos prompts

- Copiar el bloque “Prompt” tal cual o ajustar el tipo de negocio (ej. “taller de motos” vs “taller de autos”).
- En la imagen generada, **la barra superior o el rectángulo “para logo”** se deja en gris/blanco; en el PDF o en un editor se superpone el logo del cliente y, si hace falta, un overlay de color sobre el acento.

# Variantes de producto que el PDF debería poder reflejar

Listado de **opciones** que conviene tener en el flujo (diagnóstico, modal de propuesta o backend) para que el PDF y los precios no hablen solo de “un” producto genérico sino del caso concreto.

## Web

| Opción | Descripción | Qué mostrar en el PDF |
|--------|-------------|------------------------|
| **Tipo de web** | Landing (una página) / Multi‑página / Con catálogo online | Mockups que correspondan: landing con WhatsApp, multi‑página con menú de secciones, catálogo con grid de productos. Texto que diga “Incluye: landing” o “Incluye: sitio de X secciones” o “Incluye: catálogo online”. |
| **Botón WhatsApp** | Sí / No | En mockups de web, mostrar u ocultar el botón flotante según esta opción. |

## Sistemas (taller, servicio técnico, restaurante)

| Opción | Descripción | Qué mostrar en el PDF |
|--------|-------------|------------------------|
| **Un local vs varias sucursales** | Un solo local / Múltiples sucursales | Si es multi: mockup o texto que mencione “gestión por sucursal”, “selector de local”, “reportes por sucursal”. |
| **Roles y usuarios** | Solo admin / Admin + empleados (técnicos, cajero, etc.) | Si hay roles: mockup de “Usuarios y roles” (Admin, Técnico, Recepción) o texto “Diferentes perfiles: admin, técnicos, recepción”. |
| **Módulos** | Órdenes, inventario, reportes, etc. | Lista en el plan (ya se hace en parte). Reforzar en el PDF con una pantalla de dashboard o de reportes si el plan incluye métricas. |

## Combo (sistema + web)

| Qué debe quedar claro en el PDF |
|---------------------------------|
| Mostrar **ambas cosas**: al menos una vista de **web** (landing o catálogo) y al menos una vista del **sistema** (panel, órdenes, inventario). |
| Texto tipo: “Incluye sitio web profesional + panel de gestión”, “Web con catálogo + sistema de órdenes y stock”. |
| Precio del combo visible y ventaja frente a “solo sistema” o “solo web”. |

## Resumen para implementación

- En el **modal de generación de PDF** (o en el diagnóstico): poder elegir **tipo de web** (landing / multi‑página / catálogo) y **para sistemas**: un local vs varias sucursales, con o sin gestión de roles.
- En el **backend / catálogo de productos**: tener planes o flags que distingan “web landing”, “web catálogo”, “sistema un local”, “sistema multi‑sucursal”, “con roles”, “combo web + sistema”.
- En las **plantillas HTML** de cada tipo (web, taller, restaurante, servicio técnico): usar los mockups que correspondan a esas variantes (por ejemplo, si el usuario eligió “web con catálogo”, mostrar en la portada o en una página interior el mockup de catálogo; si eligió “multi‑sucursal”, incluir una línea o bloque que lo mencione y, si hay imagen, el mockup de selector de sucursal).

Con esto los prompts quedan orientados a **mockups realistas y adaptables**, y tienes el **listado de casos** para que el PDF y la lógica de negocio cubran landing, multi‑página, catálogo, un local, varios locales y roles.

# Qué falta para que la creación del PDF elija bien las imágenes

Resumen de **huecos** en el flujo actual para que la elección de imágenes sea más completa y precisa.

## 1. Opciones que no existen en el flujo

- **Tipo de web** (solo aplica si hay web en la oferta): landing / multi‑página / con catálogo.  
  Sin esto, el editor no puede mostrar labels ni hints distintos (ej. “Catálogo en escritorio – prompt 1.4”) ni sugerir los prompts correctos.
- **Tipo de sistema** (para taller, servicio técnico, restaurante): un local vs varias sucursales; con o sin roles (admin, técnicos, empleados).  
  Sin esto, no se puede sugerir “sube un mockup de selector de sucursal (5.3)” o “pantalla de usuarios y roles (3.9 / 4.5)”.
- **Combo explícito:** oferta “solo web”, “solo sistema” o “web + sistema”.  
  Sin esto, el PDF no puede adaptar texto ni slots para mostrar claramente ambos (web + panel) cuando sea combo.

## 2. Editor modal (ProposalEditorModal)

- Los **labels y hints** de cada slot son fijos por plantilla; no cambian según “tipo de web” o “tipo de sistema”.
- No hay **referencia al documento de prompts** (enlace a `mockup-prompts.md` o a una “Guía de mockups”) para que quien sube sepa qué imagen generar o qué prompt usar.
- **Genérica** solo muestra 3 slots (mockup1, 2, 3); mockup4 y mockup5 no se ofrecen. Si más adelante se agrega “web con catálogo”, harían falta al menos 4 slots para genérica (p. ej. catálogo escritorio + catálogo móvil).

## 3. Backend / datos de la propuesta

- No se guardan **tipo de web**, **tipo de sistema** ni **combo** en la propuesta ni en el DTO de generación de PDF.  
  Para que las plantillas o el editor adapten contenido, estos campos tendrían que existir y pasarse al generar el PDF.
- No hay **mapeo en código** “si tipo_web = catálogo entonces hint del slot 2 = Catálogo escritorio (prompt 1.4)”. Ese mapeo está solo en este documento; implementarlo en el front (o en un config) permitiría hints dinámicos.

## 4. Plantillas HTML

- Las plantillas **no reciben** variables como `tipoWeb` o `multiSucursal`; no pueden mostrar textos condicionales (“Incluye gestión por sucursal”, “Incluye catálogo online”) ni cambiar qué slot se usa en qué bloque.
- Para **combo**, no hay una plantilla que combine de forma explícita “página de web” + “página de sistema” con secciones y títulos diferenciados; hoy se reutiliza taller/servicio técnico que ya mezclan web y panel.

## 5. Resumen de pasos recomendados

| Paso | Dónde | Qué hacer |
|------|--------|-----------|
| 1 | Diagnóstico o modal de propuesta | Añadir campos: tipo de web (landing / multi‑página / catálogo), tipo de sistema (un local / sucursales; con/sin roles), oferta (solo web / solo sistema / combo). |
| 2 | Backend / DTO | Incluir en la propuesta o en `GeneratePdfDto` estos campos para que el servicio y las plantillas los reciban. |
| 3 | ProposalEditorModal | Mostrar labels/hints por slot según plantilla **y** variantes (tipo web, tipo sistema, combo). Incluir enlace o texto “Ver guía de mockups” apuntando a este doc o a una versión usuario. |
| 4 | proposal.service (buildVariablesForTemplate) | Pasar tipo de web, tipo de sistema y combo a las variables del template para textos condicionales. |
| 5 | Plantillas HTML | Opcional: bloques condicionales (“Si catálogo…”, “Si multi‑sucursal…”) para títulos o descripciones. |
| 6 | Catálogo de precios / planes | Alinear planes (Básico, Pro, etc.) con estas variantes para que el PDF muestre precios y textos coherentes con lo elegido. |

Con la **asignación slot ↔ prompt** de la sección anterior y esta lista de huecos, se puede hacer que la creación del PDF elija bien las imágenes (y los textos) en función del tipo de web, tipo de sistema y combo.
